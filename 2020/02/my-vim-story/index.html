My Vim Story

If you have worked in a command line at some point, I am sure you can praise the wisdom of learning how to use a text editor. On Unix/Linux command lines, text is king and learning how to manipulate text effectively goes a long way for productivity. If you know how to open a file and edit its content, that enables you to configure your system.

I have been using Linux for a long time—even on my desktop/laptop system. And my text editor of choice is called “Vim". I occasionally use other editors such as “gedit” and “UberWriter". However, most of the time when I am typing something in plain text, writing code, or editing a configuration file, I am using vim.

 

 

Like many before me, I started my editor journey by running ‘vim’, then failing to exit it, and finally resigning to reboot the computer instead. Thus, I initially became an ‘emacs’ user—primarily because ‘nano’ was not around yet. Eventually, I learned the bare minimum of vim use, just enough to do basic text editing.

Once you used notepad.exe on Windows you have used text editors—they said. Vim or not, a text editor is a text editor—I thought. But when other people who knew how to use vim used it, it still looked like magic.

So—after years of working on Linux—somewhere around 2009 I took my time to properly learn vim and customize it to my liking. I think the act of writing multiple blog posts over the years about my editor configuration is proof enough how much I ended up liking vim. It is powerful and versatile—and more importantly, I enjoy using it.

Back then my vim setup was a jumble of config files and a bunch of plugins dropped into a directory until it croaked reasonably the way I wanted. It was done out of necessity because vim did not have decent package management for plugins at that point.

 

 

When Pathogen came along, I jumped at the opportunity to organize things better. I had the idea that git submodules could be used to manage individual plugins. I looked around and there seemed to be nobody doing that at the time. Therefore, I took a day and made the plunge to use Pathogen to manage vim plugins where individual plugins were included as git submodules. To the best of my knowledge no one had publicly shared a similar setup at the time—which was a humble brag for me.

These past versions of my vim configuration files are no longer there. But a somewhat later archived version is on a git repo, while my current vimfiles are maintained on GitHub too. The most obvious change since then had been the switch to vim-plug for plugin management, which eliminated the need to juggle submodules and made the whole process quite painless.

In any case, I plan to update this post sometime to include a few section explaining my up to date vim configuration. Just as it sounds, it will not be about how good or bad vim is, nor will it be about how to use vim. When I add those sections, it will be in the hope that it will be useful to someone, just like it helped me to read what others were doing.

 

 

Latest version of my .vimrc can be found on GitHub.
